STEP 1 フレーム抽出

360度動画から全天球（equirectangular）フレームを抽出します。

⏱ 所要時間: 数分（動画の長さによる）

手順

  • 360度動画ファイル（.mp4/.mov）を用意する
  • FFmpegで全天球フレームを連番画像に抽出
  • 出力先: images_S/ フォルダ

FFmpegコマンド

1 (低) 10 10 (最高)
上のボタンでコマンドを生成
出力仕様: equirectangular 7680x3840（元動画の解像度を維持）。
命名例: 0001.jpg, 0002.jpg, ...
容量注意: 1フレーム約10-15MBになります。1000フレームで約10-15GB必要です。

ぶれフレーム除去（推奨）

動画から抽出したフレームにはぶれた画像が含まれます。 Extract_sharpest_frame（kotohibi氏）で動画から直接シャープなフレームのみ抽出するか、 抽出後にLaplacianフィルタで間引くことを推奨します。

1-A
方法A: 動画から直接シャープフレーム抽出
tools/Extract_sharpest_frame/ を使用。N フレームごとに最もシャープな1枚を自動選択。
python extract_sharpest_frame.py --video input.mp4 --chunk-size 30 --output-dir images_S --jpeg-quality 95
1-B
方法B: 抽出済みフレームのぶれ解析
Laplacianの分散でシャープネスをスコア化し、閾値以下を除外。
# 解析のみ（削除しない） python filter_blurry.py --input images_S --analyze # 閾値300以下をbackupに移動 python filter_blurry.py --input images_S --threshold 300 --move images_S_backup

STEP 2 Metashape全天球Sphericalアライメント

全天球画像のカメラ位置を推定し、camera.xmlとPLYをエクスポートします。

⏱ 所要時間: 5-15分（画像数による）   手動操作

Metashape操作手順

  • Metashapeを起動 → Workflow → Add Photos → images_S/ の全画像を追加
  • Camera Calibration → Camera Type を Spherical に変更
  • Workflow → Align Photos を実行
  • アライメント結果を確認（全画像がアライメントされているか）
  • File → Export → Export Cameras → camera.xml として保存
  • PLYもエクスポート（点群データ）

推奨設定

項目推奨値備考
精度高精度=高 でアライメント
座標事前選択推定値連番は端が合わない傾向あり、推定値を推奨
キーポイント上限40,000デフォルトより多め
タイポイント上限4,000デフォルトより多め
注意: 連番画像は端（先頭と末尾のフレーム）が合わない傾向があります。座標事前選択を「推定値」にすることを推奨します。
出力: camera.xml（カメラ位置・回転行列）+ PLY（点群データ）
Metashape camera.xmlの座標系について

Metashapeの camera.xml には各カメラの変換行列が c2w (camera-to-world) で格納されています。

  • 座標系: Z-up
  • 回転行列: c2w (camera-to-chunk)
  • 位置: chunk座標系でのカメラ中心

この情報は後続ステップのXMP生成やXML変換で使用されます。

コマンド生成

上のボタンでコマンドを生成
FFmpeg v360フィルタの詳細

FFmpegの v360 フィルタは equirectangular → flat (perspective) 変換を行います。

ffmpeg -i input.jpg -vf "v360=equirect:flat:h_fov=90:v_fov=90:yaw=0:pitch=0:roll=0,scale=1920:1920" -frames:v 1 -q:v 2 output.jpg # yaw: 水平方向の角度 (0, 45, 90, ... 315) # pitch: 垂直方向の角度 (0=水平, -30=下, +30=上) # roll: 回転 (常に0) # h_fov/v_fov: 水平/垂直視野角

テスト結果: exact モードが最も安定。initial はRSが位置を調整してスプリット（分離）する場合があります。
4-2
画像の間引き（ぶれ + 動的オブジェクト）
透視投影画像のぶれと動的オブジェクトを間引きます。XMPも同時に処理されます。

ぶれフィルタリング（自動）

# 1. シャープネス解析（まず確認） python tools/filter_blurry.py --input images_P --analyze --csv sharpness.csv # 2. 閾値以下をbackupに移動（XMPも同時） python tools/filter_blurry.py --input images_P --threshold 100 --move images_P_backup # 3. dry-runで確認してから実行 python tools/filter_blurry.py --input images_P --threshold 100 --dry-run
注意: ぶれ画像と動的オブジェクト（人物、車両など）は3DGS品質を下げます。間引いた後、対応するXMPも自動削除されます。
推奨: まず --analyze でスコア分布を確認し、適切な閾値を決めてから間引く。
座標変換の技術詳細

Metashape → RS の座標変換:

  • Metashape: Z-up, c2w, OpenCVカメラ (X右, Y下, Z前方)
  • RS XMP: Y-up, c2w → 転置して w2c を格納
  • 回転行列の変換: R_rs = [ R[0], -R[2], R[1] ]
  • 位置の変換: (x, y, z) → (x, -z, y)
⏱ 所要時間: 数分

エリア分割オプション

フレーム数が多い場合はエリアに分割してRSで処理。すべての分割パターンで2フレームのオーバーラップを入れる。

操作手順

  • エリア分割した画像フォルダをRSに読み込み（2フレームオーバーラップ付き）。XMPは自動検出される
  • Align Settings → Use Camera Metadata: Exact に設定
  • Align Photos を実行（~1分/エリア）
  • 8方向が正しく展開されているか目視確認
  • 必要に応じてMeshをHigh Detailで構築（~8分/エリア）
  • Registration を CSV でエクスポート
  • Point Cloud / Mesh を PLY でエクスポート
"initial" vs "exact": initial だとRSがカメラ位置を自由に調整するため、画像グループがスプリット（分離）する場合があります。exact は相対位置を維持するため安定です。

RSのエクスポートデータで3D Gaussian Splattingモデルを学習します。

⏱ 所要時間: 数十分（画像数・Max Gaussiansによる）

トレーニングツールの選択

Postshot（推奨）
CSV + PLY をそのまま読み込み。変換不要で最も簡単。大規模データセット（数千枚）でも安定。
推奨（大規模）
LichtFeld Studio
COLMAP形式に変換が必要。無料・オープンソース。小〜中規模（数百枚）で検証済み。
無料

Postshot: データの準備

  • RSからエクスポートした CSV（Registration）と PLY（Point Cloud）を用意
  • 画像フォルダのパスを確認
  • Postshotで Import → CSV + PLY を選択
  • 画像フォルダを指定してトレーニング開始
  • 完了後、.ply / .splat としてエクスポート
メリット: COLMAP変換が不要。座標変換ミスのリスクなし。大規模データ（6,000枚以上）でも安定動作を確認済み。

LichtFeld Studio: COLMAP形式への変換

LichtFeld Studio用にCOLMAP形式（フラット構成）で出力します。

colmap/ ├── cameras.txt ├── images.txt ├── points3D.ply ← RS mesh PLY └── images/ ← 透視投影画像

cameras.txt

# Camera list with one line of data per camera: # CAMERA_ID, MODEL, WIDTH, HEIGHT, PARAMS[] # PINHOLE: fx, fy, cx, cy 1 PINHOLE 1920 1920 960.0 960.0 960.0 960.0
FOV 90度、1920x1920の場合: f = w/2 = 960.0, cx = cy = 960.0

images.txt の変換ルール

項目変換
回転XMPの値はすでにw2c → そのまま使用（再転置しない）
平行移動 TT = -R_w2c @ C（C = CSVのx, y, alt）
クォータニオンwxyz（scalar first）
絶対にやってはいけないこと: 分割後のXMPはすでにw2c（転置済み）です。再度転置するとc2wに戻り結果が壊れます。

変換フロー

Metashape XML (c2w, Z-up) ↓ 変換: R_rs = [R[0], -R[2], R[1]], pos: (x,-z,y) XMP for RS (w2c after transpose, Z-up world) ↓ XMPの値はすでにw2c → そのまま使用 COLMAP images.txt (w2c, OpenCV) ↓ T = -R_w2c @ C, quaternion wxyz LichtFeld Studio → 3DGS
最もよくあるミス:
1. XMPの値を再転置してしまう（すでにw2cなのにさらにtransposeする）
2. 平行移動 T の計算で R_c2w を使う（正しくは R_w2c）
3. クォータニオンの順序を xyzw にする（正しくは wxyz）
