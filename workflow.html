STEP 1 フレーム抽出

360度動画から全天球（equirectangular）フレームを抽出します。

⏱ 所要時間: 数分（動画の長さによる）

手順

  • 360度動画ファイル（.mp4/.mov）を用意する
  • FFmpegで全天球フレームを連番画像に抽出
  • 出力先: images_S/ フォルダ

FFmpegコマンド

上のボタンでコマンドを生成
出力仕様: equirectangular 7680x3840（元動画の解像度を維持）。
命名例: 0001.jpg, 0002.jpg, ...
容量注意: 1フレーム約10-15MBになります。1000フレームで約10-15GB必要です。

STEP 2 Metashape Sphericalアライメント

全天球画像のカメラ位置を推定し、camera.xmlをエクスポートします。位置推定のみ行います。

⏱ 所要時間: 5-15分（画像数による）   手動操作

Metashape操作手順

  • Metashapeを起動 → Workflow → Add Photos → images_S/ の全画像を追加
  • Camera Calibration → Camera Type を Spherical に変更
  • Workflow → Align Photos を実行（デフォルト設定でOK）
  • アライメント結果を確認（全画像がアライメントされているか）
  • File → Export → Export Cameras → Spherical/camera.xml として保存
ポイント: Metashapeは位置推定（カメラの座標と向き）のみに使います。メッシュや点群はここでは生成しません。
Metashape camera.xmlの座標系について

Metashapeの camera.xml には各カメラの変換行列が c2w (camera-to-world) で格納されています。

  • 座標系: Z-up
  • 回転行列: c2w (camera-to-chunk)
  • 位置: chunk座標系でのカメラ中心

この情報はSTEP 4のXMP生成で使用されます。後続ステップではRealityCapture (Y-up) に変換する必要があります。

注意: Metashape Standard版でも360アライメントは可能です。Professional版は必要ありません。

STEP 3 FFmpeg v360 8方向分割

equirectangular画像をFFmpegのv360フィルタで8方向の透視投影（perspective）画像に分割します。

⏱ 所要時間: 数分〜数十分（画像数による）

出力仕様

項目値
方向数8方向（45度間隔: 0, 45, 90, 135, 180, 225, 270, 315度）
FOV90度
解像度1920 x 1920 px
命名規則{フレーム番号}_{方向角度}.jpg 例: 0001_000.jpg

コマンド生成

上のボタンでコマンドを生成
重要: 8方向すべて同じ中心点（カメラ位置）から撮影したものとして扱います。オフセットは入れません。
FFmpeg v360フィルタの詳細

FFmpegの v360 フィルタは equirectangular → flat (perspective) 変換を行います。

ffmpeg -i input.jpg -vf "v360=equirect:flat:h_fov=90:v_fov=90:yaw=0:pitch=0:roll=0,scale=1920:1920" -frames:v 1 -q:v 2 output.jpg # yaw: 水平方向の角度 (0, 45, 90, ... 315) # pitch: 垂直方向の角度 (常に0 = 水平) # roll: 回転 (常に0) # h_fov/v_fov: 水平/垂直視野角

バッチ処理は全フレーム x 8方向のループで実行します。

STEP 4 XMP生成（XML → XMP変換）

Metashapeのcamera.xmlから各透視投影画像のXMPサイドカーファイルを生成します。

⏱ 所要時間: 数秒

変換内容

Metashapeの c2w 行列をトランスポーズして w2c に変換し、RealityCapture用のXMP形式で書き出します。

項目値
xcr:Rotationc2w → 転置して w2c、行優先9値
PosePriorexact（相対位置を維持）
CalibrationPriorexact
中心点8方向すべて同一（オフセットなし）

PosePriorの選択肢

exact
相対位置を維持。RSが大きく動かさない。安定。
推奨
initial
RSが自由に調整可能。スプリットが起きる場合あり。
不安定な場合あり
locked
完全に固定。RSは位置を一切変更しない。
テスト結果: exact モードが最も安定。initial はRSが位置を調整してスプリット（分離）する場合があります。
座標変換の技術詳細

Metashape → RS の座標変換:

  • Metashape: Z-up, c2w, OpenCVカメラ (X右, Y下, Z前方)
  • RS XMP: Y-up, c2w → 転置して w2c を格納
  • 回転行列の変換: R_rs = [ R[0], -R[2], R[1] ]
  • 位置の変換: (x, y, z) → (x, -z, y)

変換行列 S': [[1,0,0], [0,0,-1], [0,1,0]]

  1. c2w回転にyaw回転を適用: R = R_c2w @ R_yaw(angle)
  2. row0 = R[0] そのまま
  3. row1 = -R[2] (Z-upのZ軸反転)
  4. row2 = R[1] (Z-upのY軸)
回転診断結果: Pattern I (c2w + 行1,2反転) がRSの正解データと一致。
roll差分誤差: < 0.1度 / 絶対roll誤差: < 0.8度
xcr:Rotation の格納形式

xcr:Rotation には c2w (camera-to-world) の回転行列が行優先 (row-major) で9値格納されています。

RSに渡す際は転置して w2c にする必要があります。ただし、STEP 6のCOLMAP変換時はXMP内の値はすでに w2c なので再転置は不要です。

STEP 5 RealityCapture アライメント + メッシュ

XMP付き画像をRealityCaptureに読み込み、Align → Mesh構築 → CSV + mesh PLY エクスポート。

⏱ 所要時間: Align ~1分/エリア + Mesh ~8分/エリア（24分割時）   手動操作

操作手順

  • エリア分割した画像フォルダをRSに読み込み（2フレームオーバーラップ付き）。XMPは自動検出される
  • Align Settings → Use Camera Metadata: Exact に設定
  • Align Photos を実行（~1分/エリア）
  • 8方向が正しく展開されているか目視確認
  • Mesh を High Detail で構築（~8分/エリア）
  • Registration を CSV でエクスポート（カメラ位置情報）
  • Mesh を PLY でエクスポート（密な点群として使用）
重要: Align-onlyのsparse点群ではダメ!
アライメントのみで得られるsparse点群でスプラッティングを行うと結果が崩壊します。必ずMesh PLYを使ってください。

PLYサイズ制限

Mesh PLYはfloat色+面情報を含むため、COLMAP用に点群形式（uchar色、面なし）に変換する必要があります。

エリア規模画像数mesh PLYサイズ状態
小エリア（24分割）~700枚~800MBそのまま使用可能
大エリア（2分割）~4000枚~6GBダウンサンプリング必要
PLYダウンサンプリングの詳細（大エリアの場合）

大きなmesh PLY（~6GB）は直接使用できないため、頂点をダウンサンプリングします。

サンプリング率頂点数ファイルサイズ結果
1/2~6500万~937MB失敗
1/3~4400万~625MBOK
1/4~3300万~469MBOK
上限目安: ~4000-5000万頂点、600-700MB程度が上限。
RSのCOLMAPネイティブエクスポートについて
使わないこと: RSのCOLMAPエクスポート機能は1画像しかエクスポートしません。STEP 6の手動COLMAP変換を使ってください。
"initial" vs "exact": initial だとRSがカメラ位置を自由に調整するため、画像グループがスプリット（分離）する場合があります。exact は相対位置を維持するため安定です。

STEP 6 COLMAP形式への変換

RSのCSV + mesh PLY + XMPからCOLMAP形式のファイルを生成します。

⏱ 所要時間: 数秒

出力ファイル構成

LichtFeld Studio用にフラット構成（sparse/0/ は不要）で出力します。

colmap/ ├── cameras.txt ├── images.txt ├── points3D.ply ← RS mesh PLY (hardlink) └── images/ ← 透視投影画像 (symlink or copy)

cameras.txt

# Camera list with one line of data per camera: # CAMERA_ID, MODEL, WIDTH, HEIGHT, PARAMS[] # PINHOLE: fx, fy, cx, cy 1 PINHOLE 1920 1920 960.0 960.0 960.0 960.0
FOV 90度、1920x1920の場合: f = w/2 = 960.0, cx = cy = 960.0

images.txt の変換ルール

項目変換
回転XMPの値はすでにw2c → そのまま使用（再転置しない）
平行移動 TT = -R_w2c @ C（C = CSVのx, y, alt）
クォータニオンwxyz（scalar first）
形式IMAGE_ID QW QX QY QZ TX TY TZ CAMERA_ID NAME
絶対にやってはいけないこと: 分割後のXMPはすでにw2c（転置済み）です。ここで再度転置するとc2wに戻ってしまい、結果が壊れます。
points3D.ply について

RS mesh PLYをそのままhardlinkで points3D.ply として配置します。

  • Mesh PLY: float色 + faces → 点群形式（uchar色、facesなし）に変換が必要
  • Align-onlyのsparse点群は使わない（スプラッティングが崩壊する）
  • 大きいPLYはダウンサンプリング（STEP 5参照）
COLMAP images.txt の座標系について

COLMAP images.txt の座標系:

  • w2c（world-to-camera）
  • OpenCV座標系: X-right, Y-down, Z-forward
  • クォータニオン: wxyz（scalar first）

各画像は2行で記述されます:

# 1行目: IMAGE_ID QW QX QY QZ TX TY TZ CAMERA_ID IMAGE_NAME # 2行目: POINTS2D[] as (X, Y, POINT3D_ID) - 空行でOK 1 0.9239 0.0 0.3827 0.0 1.234 -0.567 2.345 1 0001_000.jpg

GUT (Geometric Undistortion Transform) について

GUTはレンズ歪み補正のための機能です。equirectangularを直接LF Studioに入れる場合は必要ですが、今回のワークフローではPINHOLEモデル（v360で分割済み）を使うため不要です。

GUTを有効にすると処理が重くなるだけで、PINHOLEには効果がありません。

equirectangularを直接使う方法（非推奨）

龍 lilea (lileaLab) がequirectangularを直接LF Studioに入れる方法をテストしていますが、GUTが重く処理時間が大幅に増加します。

本ワークフローのv360分割 → PINHOLE方式の方が効率的です。

最もよくあるミス:
1. XMPの値を再転置してしまう（すでにw2cなのにさらにtransposeする）
2. 平行移動 T の計算で R_c2w を使う（正しくは R_w2c）
3. クォータニオンの順序を xyzw にする（正しくは wxyz）
